# Contributing

Thank you for your interest in contributing to this project. This document outlines the contribution workflow, coding standards, and available community support.

## Contributor License Agreement

Most contributions require you to agree to a Contributor License Agreement (CLA), confirming that you have the right to grant us permission to use your contribution.

For details, please visit [Contributor License Agreement](https://github.com/zhejianglab/.github/blob/main/CONTRIBUTOR_LICENSE_AGREEMENT.md).

## Ways to Contribute

Contributions are welcome in many forms, including but not limited to:

- Reporting bugs, issues, or unexpected behaviors
- Suggesting new features or improvements
- Improving documentation, tutorials, or examples
- Contributing code (new features, bug fixes, refactoring)
- Providing benchmarks, evaluation scripts, or example datasets
- Sharing best practices for domain LLM training

## Reporting Issues

If you encounter a bug or have a feature suggestion:

- Please check existing issues to avoid duplication.
- Open a new issue and include:
   - A clear description of the problem or proposal
   - Steps to reproduce the issue (if applicable)
   - Expected behavior and actual behavior
   - Relevant logs, configurations, or screenshots (if applicable)

Clear and detailed reports help us respond more efficiently.

## Submitting Pull Requests

We use Pull Requests (PRs) to review and merge contributions.

### Recommended Workflow

- Fork the repository and create a new branch from `main`
- Make changes in small, logical, and well-documented commits
- Ensure your code:
   - Follow existing code structure and style
   - Does not break existing functionality
- Run relevant tests or provide validation results if applicable
- Submit a Pull Request with:
   - A clear summary of what the PR does
   - The motivation and context of the contribution
   - References to related issues (if any)

## Code Style and Quality

- Follow existing coding conventions and project structure
- Write clear comments for non-trivial logic
- Prefer clarity and reproducibility over over-optimization

## Documentation Contributions

Documentation is as important as code. You are welcome to contribute:

- User guides and tutorials
- Domain-specific usage examples 
- Hardware and accelerator adaptation notes
- Reproducible experiment or deployment instructions

Please ensure documentation changes are clear, concise, and aligned with the project structure.

## Community Support & Incentives

To encourage high-quality contributions and foster an active open-source ecosystem, we provide the following support to selected contributors:

- Compute Resource Support

   Contributors with impactful contributions (e.g., high-quality code, datasets, or reproducibility studies) may apply for access to shared GPU resources for experimentation, training, or benchmarking.

- Early Access & Collaboration Opportunities

   Active contributors may gain early access to new features, internal tools, or datasets, and may be invited to participate in joint research or engineering collaborations.

- Community Recognition

   We value all contributions and provide recognition in the following ways:

   - Contributor acknowledgements in the repository
   - Highlighting major contributions in release notes
   - Potential co-authorship in technical reports or papers
